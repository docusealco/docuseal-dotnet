# docuseal-dotnet

.NET SDK for the DocuSeal API. NSwag generates the entire client here:
its output is idiomatic .NET (typed optional query parameters,
CancellationToken on every method, System.Text.Json, IDocusealClient
interface), so only a thin partial extension is written by hand. This
differs from docuseal-js/go/java (hand-written client + generated models)
and from docuseal-ruby/python/php (fully hand-written).

## Architecture

- `src/Docuseal/DocusealClient.g.cs` - generated by NSwag (openapi2csclient),
  never edit by hand. Regenerate with `./generate-types.sh
  [spec-path-or-url]`. Contains the full async client, DTOs and
  `DocusealException`. Method names come from spec operationIds
  (`GetTemplatesAsync`, `UpdateSubmitterAsync`).
- `src/Docuseal/DocusealClient.cs` - hand-written partial extension:
  constructors with API key and base URL (Global/EU/on-premises),
  `PrepareRequest` hook adding `X-Auth-Token` and `User-Agent`, and
  `CreateSubmissionAsync` against `POST /submissions/init`, which is not in
  the public spec (the generate script drops the legacy `POST /submissions`
  so this name is free).
- `generate-types.sh` preprocesses the spec with ruby: drops webhook
  schemas and the legacy `POST /submissions`, and extracts inline object
  schemas into components with deterministic path-derived names - otherwise
  NSwag names them `Fields2`/`Anonymous` with unstable counters. NSwag
  consumes the OpenAPI 3.1 spec directly.

## Source of truth

The OpenAPI spec is generated by `lib/api_spec/` in the docuseal-website
repository and served at https://console.docuseal.com/openapi.yml. Do not
fix type issues here - fix them in the spec and regenerate.

## Conventions

- No external package dependencies: System.Text.Json only.
- Build: `dotnet build src/Docuseal` (local .NET lives in `~/.dotnet` if
  not installed system-wide). No tests yet.
